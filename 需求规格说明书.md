# 《雀友聚》麻将拼团网站需求规格说明书 (SRS)

## 1. 引言

### 1.1 目的

本文档旨在详细描述"雀友聚"拼团网站的功能需求、业务逻辑及用户交互流程，作为开发、测试及验收的依据。

### 1.2 项目背景

针对棋牌爱好者"凑人难"、"信息不对称"的痛点，通过LBS（基于位置的服务）和社交审核机制，建立一个透明、高效、有信用约束的同城约局平台。

---

## 2. 角色定义

- **游客：** 未登录用户，仅可浏览各城市社区概况，无法查看地点详情和发起申请。
- **注册用户：** 经邮箱验证的用户，可发起拼团、申请加入、进行评价。
- **房主：** 发起拼团的用户，拥有管理成员（审核、踢人）的权限。
- **管理员：** 后台管理人员，负责城市配置、违规处理及申诉仲裁。

## 4. 用户场景 (User Scenarios)

### 4.1 约局全流程场景

**场景描述：** 三个麻将爱好者通过平台成功拼局并完成对局的全过程

| 步骤 | 角色  | 动作描述                                                              | 系统响应                                           |
| ---- | ----- | --------------------------------------------------------------------- | -------------------------------------------------- |
| 1    | 房主A | 打开APP，定位到北京，创建拼团：今晚20:00，地点为"朝阳公园茶馆"，4人局 | 创建成功，状态为"招募中"                           |
| 2    | 用户B | 浏览拼团列表，看到房主A的拼团，距离1.2km，信用分95                    | 可查看详情                                         |
| 3    | 用户B | 点击"申请加入"                                                        | 系统校验用户B是否有未结束的团，发送申请通知给房主A |
| 4    | 房主A | 收到申请，查看用户B的信用分和评价历史                                 | 显示用户B过往评价摘要                              |
| 5    | 房主A | 点击"同意"                                                            | 用户B加入成功，拼团状态更新为"3/4"                 |
| 6    | 用户C | 同样申请并加入                                                        | 拼团状态更新为"满员"                               |
| 7    | 房主A | 20:00前，在聊天室发起群聊，分享导航位置                               | 成员收到通知                                       |
| 8    | 全员  | 到达现场，开始对局                                                    | 系统开始记录时间                                   |
| 9    | 全员  | 对局结束，房主A结束拼团                                               | 触发评价流程                                       |
| 10   | 全员  | 互相评价（准时、态度、牌品）                                          | 信用分+1分，评价记录入库                           |

### 4.2 紧急补位场景

**场景描述：** 拼团前临时有人退出，房主使用紧急补位功能

| 步骤 | 角色  | 动作描述                         | 系统响应                               |
| ---- | ----- | -------------------------------- | -------------------------------------- |
| 1    | 房主A | 原定拼团满员，距离开始还有30分钟 | 状态为"满员"                           |
| 2    | 成员B | 因事临时退出拼团                 | 拼团状态变更为"3/4"                    |
| 3    | 房主A | 收到成员退出通知，点击"设置加急" | 校验：房主信用分=100，今日未设置过加急 |
| 4    | 房主A | 确认设置加急                     | 拼团获得"加急"标签，置顶显示在列表首位 |
| 5    | 用户D | 浏览拼团列表，优先看到加急拼团   | 加急拼团带有特殊标记                   |
| 6    | 用户D | 快速申请加入                     | 拼团状态恢复"满员"                     |

### 4.3 信用勋章获得场景

**场景描述：** 用户通过良好行为获得各类信用勋章

| 勋章类型 | 获得条件                        | 展示效果                     |
| -------- | ------------------------------- | ---------------------------- |
| 守时标兵 | 连续10次准时到场（评价均为5星） | 用户头像旁显示"守时标兵"徽章 |
| 金牌房主 | 累计组织10场且平均评价≥4.5星    | 个人主页显示"金牌房主"徽章   |
| 从不鸽子 | 连续10场从未触发爽约扣分        | 信用分旁显示"信誉极佳"标签   |
| 已实名   | 通过第三方实名校验              | 显示"已认证"蓝V标识          |

### 4.4 雀友俱乐部场景

**场景描述：** 小区固定牌友创建俱乐部，内部约局

| 步骤 | 角色        | 动作描述                 | 系统响应                        |
| ---- | ----------- | ------------------------ | ------------------------------- |
| 1    | 用户A       | 创建"阳光小区麻将俱乐部" | 创建成功，用户A成为俱乐部管理员 |
| 2    | 用户A       | 邀请10位邻居加入俱乐部   | 成员收到邀请通知                |
| 3    | 俱乐部成员B | 发布俱乐部内部拼团       | 仅俱乐部成员可见                |
| 4    | 俱乐部成员C | 在俱乐部Tab查看内部拼团  | 显示所有俱乐部内发起的拼团      |
| 5    | 俱乐部成员D | 申请加入内部拼团         | 房主直接同意，无需审核          |

### 4.5 聊天室记分场景

**场景描述：** 使用聊天室内置记分器解决算账问题

| 步骤 | 角色  | 动作描述                         | 系统响应                  |
| ---- | ----- | -------------------------------- | ------------------------- |
| 1    | 房主A | 打开聊天室记分器，添加4位玩家    | 记分面板初始化            |
| 2    | 房主A | 每轮结束后输入每位玩家的输赢金额 | 累计计算总额              |
| 3    | 用户B | 查看自己应付款                   | 显示"需支付给房主A：50元" |
| 4    | 房主A | 对局结束，点击"生成账单"         | 自动生成对局账单汇总      |
| 5    | 房主A | 点击"分享到聊天室"               | 账单以图片形式发送到群聊  |
