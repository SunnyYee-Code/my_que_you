# 前端开发计划

## 1. 文档目标与依据
- 目标：基于仓库当前实现与《需求规格说明书》整理前端开发范围、模块拆分、状态与交互方案、接口依赖和增量实现建议，为后续迭代排期与联调提供依据。
- 依据：`需求规格说明书.md`、当前 `src/` 目录结构、已实现页面与 hooks 命名。
- 适用范围：Web 端 React 前端（当前仓库为 Vite + React + TypeScript 技术栈）。

## 2. 当前前端范围概述

### 2.1 已有前端能力（按 SRS 与代码结构归纳）
当前前端已覆盖的核心业务链路主要包括：
- 账户与基础资料：注册登录、邮箱/手机号验证码注册、城市定位、资料编辑、新用户引导、信用分申诉。
- 拼团主流程：创建拼团、城市社区列表、筛选过滤、拼团详情、申请加入、房主审核、成员退出、房主取消拼团。
- 沟通与关系链：群聊、好友系统、私信聊天、通知中心、房主邀请好友。
- 信用与安全：评价体系、举报、违禁词过滤、部分账号注销能力。
- 后台能力：管理后台、通知与实时订阅。

### 2.2 当前前端边界判断
结合仓库结构，当前前端更偏向“业务功能已跑通，但体验与扩展层仍需系统化整理”的状态，具体表现为：
- 页面已覆盖主流程，但部分需求处于“部分实现”或“能力存在但未闭环”。
- 社交化、信任增强、工具化功能仍缺页面入口、状态模型或统一交互规范。
- 实时通知、聊天室、群组状态变更等功能存在跨页面联动，需要进一步统一状态管理策略。
- SRS 中的部分描述与当前实现状态存在偏差，前端文档需要明确“已上线 / 部分实现 / 待补齐”的真实边界，避免排期误判。

## 3. 页面 / 模块拆分建议

### 3.1 一级业务模块拆分
建议按照用户心智与路由结构，将前端拆分为以下一级模块：

1. **认证与用户初始化模块**
   - 登录 / 注册
   - 邮箱 / 手机验证码验证
   - 新用户引导
   - 资料补全
   - 城市定位与首次进入逻辑

2. **拼团社区模块**
   - 首页拼团列表
   - 列表筛选与排序
   - 拼团卡片
   - 拼团详情
   - 地图信息展示
   - 紧急补位与置顶展示（待实现）

3. **拼团创建与管理模块**
   - 创建拼团表单
   - 常用地点管理（待实现）
   - 房主审核中心
   - 成员管理
   - 取消拼团 / 结束拼团
   - 房主邀请好友

4. **沟通与消息模块**
   - 群聊聊天室
   - 私信会话
   - 系统通知中心
   - 聊天工具扩展（语音 / 定位 / 记分器 / 番数计算器，待实现）

5. **信用、安全与个人中心模块**
   - 个人主页 / 设置
   - 评价记录
   - 信用分与勋章
   - 实名认证（待实现）
   - 黑名单（待实现）
   - 举报与申诉
   - 账号注销

6. **社区社交模块**
   - 雀友圈 feed（待实现）
   - 俱乐部列表 / 详情 / 成员管理（待实现）
   - 战绩与排行榜（待实现）

7. **管理后台模块**
   - 城市配置
   - 举报处理
   - 申诉裁决
   - 违禁词管理
   - 实名审核（若采用人工审核方案）

### 3.2 页面层级建议
建议按“路由页面 + 业务容器 + 展示组件 + hooks + service”组织：
- `pages/`：路由级页面，如首页、详情、创建页、设置页、聊天室页。
- `modules/`：按业务域组织容器组件，如 `group`, `chat`, `profile`, `club`, `feed`。
- `components/`：通用 UI 组件与业务复用组件。
- `hooks/`：数据请求、缓存、实时订阅、表单逻辑。
- `services/`：后端 API / Supabase 调用封装。
- `store/`：全局状态，如用户信息、城市、通知计数、会话上下文。
- `types/`：接口 DTO、领域模型、状态枚举。

### 3.3 优先级拆分建议
建议按三期推进：
- **P0（稳定主链路）**：排他性边界、状态锁定、自动结束状态同步、成员退出后的详情刷新、异常态提示统一。
- **P1（补齐高价值缺口）**：实名认证、黑名单、常用地点、紧急补位、聊天室定位消息。
- **P2（增强留存与社交）**：雀友圈、俱乐部、战绩榜、记分器、番数计算器、信用勋章。

## 4. 核心交互与状态管理建议

### 4.1 关键状态域划分
建议前端状态按以下维度拆分：
- **会话状态**：登录态、用户身份、实名认证状态、是否完成 onboarding。
- **城市上下文状态**：当前城市、定位权限、手动切换城市、本地缓存。
- **拼团状态**：列表数据、筛选条件、详情数据、申请状态、成员状态、房主操作状态。
- **消息状态**：通知未读数、群聊消息流、私信会话列表、实时连接状态。
- **个人中心状态**：资料、信用分、勋章、黑名单、申诉记录。
- **社区状态**：动态 feed、俱乐部、排行榜、工具面板状态。

### 4.2 状态管理方案建议
基于当前 React 技术栈，建议采用“服务端状态与客户端状态分离”的方案：
- **服务端状态**：优先使用 TanStack Query（若当前未全量接入，建议逐步迁移），负责列表查询、详情缓存、失效重取、乐观更新。
- **客户端轻状态**：使用 Context / Zustand 管理城市、登录态补充信息、全局弹窗、筛选面板可见性、聊天室临时输入态。
- **表单状态**：创建拼团、资料编辑、实名认证申请等表单建议统一使用 React Hook Form + Zod（或仓库现有校验方案）管理。
- **实时状态**：通知、聊天、房主审核列表建议通过 Supabase Realtime 订阅后，统一写入 query cache 或事件总线，避免页面各自监听导致重复订阅。

### 4.3 核心交互建议
1. **拼团列表 → 详情 → 申请加入**
   - 列表卡片应直接展示：时间、距离、缺口人数、房主信用分、玩法标签、是否加急。
   - 申请按钮点击前先做前端预校验：登录态、是否已在其他未结束拼团、是否被房主拉黑、是否已申请过。
   - 提交后按钮进入 loading + disabled，避免重复点击。
   - 结果反馈建议采用 toast + 局部状态刷新，而不是整页刷新。

2. **房主审核请求**
   - 房主审核列表应区分待处理 / 已通过 / 已拒绝 / 已过期。
   - 同意 / 拒绝操作建议做 optimistic UI，但必须在失败时回滚。
   - 若拼团已满员或已取消，应统一弹出不可操作提示并刷新列表。

3. **成员退出 / 房主取消 / 自动结束**
   - 这三类操作均会影响多个页面：列表、详情、聊天、通知。
   - 建议定义统一的 group status event 映射，前端收到状态变更后按 groupId 精准失效缓存。

4. **聊天室体验**
   - 先保证文字消息、图片/定位扩展入口、系统消息（谁加入/退出/被通过）统一展示。
   - 对语音、记分器、番数计算器使用抽屉式工具面板，避免聊天室主视图过载。

5. **实名认证 / 黑名单**
   - 实名与屏蔽属于强约束能力，前端需在列表、详情、私信、申请入口多个位置统一拦截，不应只在单页面判断。

### 4.4 错误态与空态建议
- 列表为空：区分“当前城市暂无拼团”和“筛选条件过严”。
- 详情失效：区分“拼团已取消”“无权限查看”“数据加载失败”。
- 聊天异常：区分“会话不存在”“你已不在该拼团中”“实时连接中断”。
- 认证失败：明确展示失败原因、重试方式和人工申诉入口。

## 5. 与后端接口的依赖点

### 5.1 已知后端依赖类型
从 SRS 和现有命名看，前端主要依赖以下后端能力：
- **认证能力**：注册、登录、验证码发送与校验、用户资料查询。
- **用户资料能力**：昵称唯一性校验、头像上传、信用分、评价摘要、实名认证状态。
- **拼团能力**：创建、列表查询、详情查询、申请加入、审核、退出、取消、结束、状态流转。
- **消息能力**：群聊、私信、通知、实时订阅。
- **关系链能力**：好友、邀请、黑名单。
- **治理能力**：举报、申诉、管理后台配置、违禁词、实名审核。
- **社区能力**：动态 feed、俱乐部、排行榜、战绩、记分工具（待建设）。

### 5.2 前端联调重点接口
建议优先梳理并稳定以下接口契约：
1. **拼团列表接口**
   - 字段：城市、距离、状态、缺口人数、加急标记、房主信用分、标签、是否可申请。
   - 风险：距离计算口径、加急排序规则、黑名单过滤口径需后端明确。

2. **拼团详情接口**
   - 字段：房主信息、成员列表、地点信息、聊天入口、申请状态、评价摘要。
   - 风险：游客 / 非成员 / 被屏蔽用户的可见字段范围需明确。

3. **申请与审核接口 / RPC**
   - 风险：排他性原则、并发申请、满员瞬时竞争、房主取消与成员退出的竞态条件。
   - 建议：接口返回明确错误码，前端按错误码做精确提示。

4. **通知与实时订阅**
   - 事件类型：申请提交、审核通过/拒绝、成员退出、拼团取消、聊天室消息、评价提醒。
   - 风险：重复消息、乱序消息、离线重连后的补偿策略。

5. **实名认证与黑名单接口（待实现）**
   - 实名认证：提交材料、审核状态、失败原因、再次提交。
   - 黑名单：新增、移除、列表查询、跨模块可见性控制。

6. **社交扩展接口（待实现）**
   - 雀友圈：发布、删除、点赞、评论、图片上传。
   - 俱乐部：创建、加入、审核、俱乐部内拼团列表。
   - 战绩榜 / 记分器：对局记录、统计聚合、权限控制。

### 5.3 接口契约建议
- 所有关键写操作返回统一结构：`success`、`code`、`message`、`data`。
- 所有“不可操作”场景提供稳定错误码，如：`GROUP_FULL`、`USER_BLOCKED`、`HAS_ACTIVE_GROUP`、`KYC_REQUIRED`。
- 时间字段统一使用 ISO 字符串，前端统一转换展示，避免时区误差。
- 图片 / 文件上传返回标准资源对象，便于聊天、动态、头像等场景复用。

## 6. 未实现需求对应的前端实现建议

### 6.1 实名认证（KYC）
- 页面建议：`/settings/verification`。
- 交互建议：展示认证说明、资料上传、审核进度、失败原因、重新提交入口。
- 组件建议：身份证上传组件、人脸提示组件、审核状态卡片。
- 状态建议：`unverified / pending / approved / rejected` 四态明确区分。
- 联动点：个人主页勋章、申请加入前校验、房主筛选实名成员能力（如后续有）。

### 6.2 黑名单屏蔽
- 页面建议：设置页新增“黑名单管理”。
- 交互建议：支持从用户卡片、私信页、详情页直接拉黑/取消拉黑。
- 前端控制点：
  - 列表中过滤对方拼团；
  - 详情页禁止查看或弱化展示；
  - 私信入口不可达；
  - 申请加入按钮直接禁用并说明原因。
- 风险：若仅做前端过滤而后端未做数据层过滤，会导致绕过，因此必须前后端双重控制。

### 6.3 常用地点收藏
- 页面建议：创建拼团页内增加“选择常用地点”弹层，设置页增加地点管理入口。
- 交互建议：创建成功后可勾选“保存为常用地点”；支持重命名、删除、置顶。
- 状态建议：地点列表本地缓存 + 服务端同步，提升表单回填速度。

### 6.4 紧急补位
- 入口建议：拼团详情页房主操作区。
- 展示建议：列表卡片增加“加急”视觉标签，置顶区块与普通排序分离。
- 校验建议：前端先校验信用分 100、当天未使用、当前团为缺员状态；后端再做最终校验。
- 反馈建议：设置成功后同步通知成员与潜在候选用户（若有推荐逻辑）。

### 6.5 聊天室工具扩展
1. **定位消息**：
   - 聊天输入区增加“发送位置”能力；
   - 消息卡片可直接跳转地图；
   - 注意仅成员可见，且不展示过精确坐标。
2. **语音消息**：
   - 需要录音权限提示、时长限制、播放状态管理。
3. **记分器 / 电子账单**：
   - 以聊天室内工具面板承载；
   - 支持对局内多轮录入、结束后生成结算摘要；
   - 账单结果可一键发送到群聊。
4. **番数计算器 / 规则查询**：
   - 建议先做规则查询，再做复杂计算；
   - 若后续引入拍照识别，应单独拆为实验性功能。

### 6.6 雀友圈 / 俱乐部 / 战绩榜
- **雀友圈**：采用独立 Tab，内容流卡片复用头像、图片宫格、互动条；支持发布、删除、评论、点赞。
- **俱乐部**：采用“列表页 + 详情页 + 管理弹层”结构；俱乐部内拼团与公共拼团切换展示。
- **战绩榜**：建议先做个人战绩页，再做好友排行，最后扩展到城市排行，避免一次性做复杂榜单。

## 7. 技术风险与实施建议

### 7.1 主要技术风险
1. **状态分散导致联动不一致**
   - 列表、详情、通知、聊天若各自维护状态，容易出现加入成功后页面不同步。

2. **实时事件与普通请求混用导致竞态**
   - 审核、退出、取消、自动结束都可能与实时订阅同时触发，容易造成重复渲染或错误覆盖。

3. **业务规则前后端不一致**
   - 排他性原则、黑名单过滤、实名门槛、加急资格若前后端判断不一致，会直接影响用户信任。

4. **扩展功能侵入主流程**
   - 雀友圈、俱乐部、记分器等新功能如果直接耦合现有页面，容易让首页和聊天室变得臃肿。

5. **性能与移动端体验风险**
   - 列表卡片信息较多、实时消息频繁、图片上传场景增多后，首屏速度与弱网体验可能下降。

### 7.2 实施建议
- **先补齐领域模型**：统一 group、request、review、notification、club、feed 等类型定义与状态枚举。
- **先收敛数据流再扩功能**：优先把拼团主链路的 query key、缓存失效、实时事件映射整理清楚。
- **建立统一错误码映射表**：前端所有写操作共用错误提示层，减少重复判断。
- **引入模块化目录约束**：新增需求按业务域落到独立模块，不继续堆在页面文件中。
- **为关键流程补前端回归用例**：至少覆盖注册、创建拼团、申请加入、房主审核、成员退出、评价、通知联动。
- **分阶段灰度未实现功能**：实名认证、黑名单、紧急补位优先灰度；雀友圈、俱乐部、记分器可在主链路稳定后上线。

## 8. 建议的前端实施顺序
1. 梳理主流程状态模型与接口错误码。
2. 修复部分实现与边界问题：排他性、状态锁定、自动结束同步、退出后的详情刷新。
3. 补齐高优先级缺失能力：实名认证、黑名单、常用地点、紧急补位。
4. 增强聊天室工具能力：定位消息、记分器基础版。
5. 拓展社区社交能力：雀友圈、俱乐部、战绩榜。
6. 完成性能优化、埋点监控、回归测试与可观测性补齐。
