# 趣友后端开发计划

## 1. 文档目标

本文档用于沉淀 `SunnyYee-Code/my_que_you` 当前阶段的后端建设方案，明确：

- 当前项目已具备的后端能力边界
- 后续后端开发的阶段目标与优先级
- 核心数据模型、接口/服务职责、任务链路与风控要求
- 联调、自测、上线与回滚的执行基线

本文档面向技术 Leader、前端开发、测试工程师与后续接手的后端同学，作为需求拆解、开发排期和验收对齐依据。

---

## 2. 项目现状概览

结合仓库现状、`README.md`、`需求规格说明书.md` 与 `src/integrations/supabase/types.ts`，可以确认本项目当前属于 **前端 + Supabase BaaS 驱动的轻后端架构**，核心特征如下：

### 2.1 技术现状

- 前端技术栈：Vite + React + TypeScript
- 数据与认证底座：Supabase
- 已存在的后端载体：
  - Supabase Auth（登录/注册能力）
  - Postgres 数据表
  - 行级权限（RLS）与数据库约束（需继续核查与补齐）
  - 由前端直接调用的数据库读写链路
- 仓库内暂未发现独立 Node.js / Python 服务目录，说明当前“后端”主要依赖 Supabase 数据层与前端直连模式

### 2.2 已识别的核心业务对象

从 `types.ts` 可识别出以下主要数据域：

- 用户域：`profiles`、`user_roles`、`credit_history`、`user_daily_actions`
- 组局域：`groups`、`group_members`、`join_requests`、`group_invitations`、`group_member_exits`
- 社交域：`friendships`、`direct_messages`、`messages`
- 平台治理域：`notifications`、`reports`、`reviews`、`banned_words`、`system_settings`、`ip_rate_limits`
- 基础数据域：`cities`、`play_styles`
- 安全辅助域：`email_otp_codes`、`deleted_emails`

### 2.3 当前架构风险

当前以前端直连 BaaS 为主的方式可以快速交付，但随着业务复杂度提升，会逐步暴露以下问题：

1. **复杂业务规则分散在前端**：例如组局人数校验、加入审批、信用分变更、举报审核等，若大量放在前端实现，容易被绕过。
2. **并发一致性风险**：抢最后名额、重复申请、重复退出、消息未读状态更新等场景需要事务或原子操作保障。
3. **风控能力不足**：频控、敏感词、黑名单、异常行为识别若只在前端处理，无法形成有效防线。
4. **可观测性不足**：缺少统一日志、错误追踪、任务执行记录与审计链路。
5. **接口契约不稳定**：前端直接依赖表结构，后续一旦数据模型调整，联动影响较大。

因此，后续需要从“前端直连数据库”逐步演进为“**数据库能力 + 受控服务层 + 任务/风控脚本**”的后端体系。

## 3. 后端建设目标

## 3.1 总体目标

建设一个围绕“组局、加入、聊天、通知、治理”闭环的稳定后端体系，满足以下要求：

- **正确性**：关键业务规则必须在服务端强校验
- **一致性**：名额、申请、审批、信用分等关键数据具备原子性保障
- **安全性**：认证、授权、频控、敏感词、举报审核链路可控
- **可维护性**：前端不直接耦合复杂业务规则，后端职责边界清晰
- **可观测性**：具备错误日志、审计记录、任务执行结果和基础监控
- **可扩展性**：为后续接入独立 Node.js / Python 服务、定时任务、运营脚本预留空间

## 3.2 本阶段交付目标

结合当前仓库现状，建议后端开发分为三个阶段推进：

### 阶段 A：后端规则收口（短期）

目标：把关键业务规则从前端收口到数据库函数 / Edge Functions / 受控接口中。

优先覆盖：

- 创建组局
- 申请加入组局
- 审批加入申请
- 退出组局
- 发送群聊消息 / 私聊消息
- 生成通知
- 举报提交

### 阶段 B：治理与风控增强（中期）

目标：补齐平台治理与风控能力。

优先覆盖：

- 敏感词校验
- 用户频控
- 信用分变更规则
- 举报审核流
- 黑名单 / 封禁 / 权限限制
- 审计日志与异常告警

### 阶段 C：服务化演进（中长期）

目标：将复杂逻辑从纯 BaaS 进一步演进为可维护的后端服务。

方向包括：

- Node.js BFF / API 服务
- Python 数据任务 / 风控脚本 / 审核辅助脚本
- 定时任务中心
- 更完善的监控、告警与运营后台支撑

## 4. 推荐后端架构

## 4.1 架构原则

1. **简单查询优先走 RLS + 视图/表**
2. **复杂写操作统一走服务端函数或受控接口**
3. **有副作用的流程必须具备事务性与幂等性**
4. **风控与治理逻辑不得只依赖前端**
5. **通知、信用分、审核等跨域流程尽量事件化处理**

## 4.2 推荐分层

### L1：数据层（Supabase Postgres）

职责：

- 核心业务表存储
- 索引、约束、唯一键、外键
- RLS 权限控制
- SQL function / RPC / trigger
- 审计辅助表

### L2：受控后端能力层

建议优先采用以下两类方式承接：

- **Supabase Edge Functions**：适合承接需要鉴权、参数校验、聚合写入、外部通知调用的接口
- **数据库 RPC / 存储过程**：适合承接强事务、强一致性写操作

推荐分工：

- 强事务链路：优先放 DB function / transaction
- 需要调用外部服务、发送消息、复杂鉴权：优先放 Edge Functions

### L3：任务与脚本层

建议后续补充：

- Python：风控扫描、数据修复、批处理、统计报表
- Node.js：定时通知、状态回收、消息补偿、后台运营脚本

### L4：前端消费层

前端只负责：

- 参数收集
- 结果展示
- 乐观 UI（如有）
- 错误提示

前端不负责：

- 最终业务规则裁定
- 名额竞争结果判定
- 信用分最终变更
- 审核结果生成

## 5. 核心业务域开发计划

## 5.1 用户与认证域

### 目标

建立可靠的用户身份、资料、权限和行为限制体系。

### 涉及表

- `profiles`
- `user_roles`
- `deleted_emails`
- `email_otp_codes`
- `user_daily_actions`
- `ip_rate_limits`

### 开发项

1. **资料完整性校验**
   - 新用户首次登录后补齐昵称、城市、偏好等信息
   - 昵称唯一性 / 合规性校验

2. **用户状态控制**
   - `is_banned`
   - `can_create_group`
   - `can_join_group`
   - 每日创建/加入次数限制

3. **认证与验证码治理**
   - OTP 发送频率限制
   - OTP 失效时间控制
   - 同邮箱 / 同 IP 限流

4. **角色体系**
   - 普通用户
   - 管理员 / 审核角色
   - 后续支持运营角色扩展

### 验收标准

- 被封禁用户无法继续创建或加入组局
- 达到日限额用户不能绕过前端限制重复操作
- 验证码接口具备频控与过期校验

---

## 5.2 组局域

### 目标

确保组局创建、展示、加入、审批、退出全流程稳定可控。

### 涉及表

- `groups`
- `group_members`
- `join_requests`
- `group_invitations`
- `group_member_exits`
- `cities`
- `play_styles`

### 核心规则

1. 只有满足资格的用户才能创建组局
2. 组局人数上限必须由后端校验
3. 同一用户不能重复加入同一组局
4. 组局状态与时间必须合法
5. 审批通过时要原子化占用名额
6. 主理人退出/取消组局需定义后续规则
7. 组局结束后状态需要自动流转

### 重点开发项

#### A. 创建组局接口

输入建议：

- 城市
- 开始/结束时间
- 地址
- 玩法标签
- 备注
- 需要人数
- 经纬度（可选）
- 是否公开

服务端必须校验：

- 用户是否允许创建组局
- 时间区间是否合法
- 人数范围是否合法
- 城市与玩法是否为有效枚举/数据项
- 日创建次数是否超限

#### B. 申请加入组局接口

服务端必须校验：

- 用户是否允许加入
- 组局是否仍可加入
- 是否已是成员
- 是否已有待审批申请
- 是否命中频控/风控规则

输出结果应明确区分：

- 已提交申请
- 需要审批
- 直接加入成功（若业务允许）
- 因名额/状态/权限失败

#### C. 审批加入申请接口

这是最关键的事务链路之一，建议使用事务/RPC 实现。

服务端步骤：

1. 锁定申请记录
2. 校验操作者是否为 host 或管理员
3. 校验组局状态是否仍可加入
4. 校验剩余名额
5. 更新申请状态
6. 插入 `group_members`
7. 写入 `notifications`
8. 必要时记录 `credit_history` 或审计日志

#### D. 退出组局接口

需明确区分：

- 普通成员退出
- 主理人解散组局
- 主理人转移
- 活动开始后是否允许退出

建议同时记录：

- 退出原因
- 退出时间
- 是否影响信用分

### 验收标准

- 并发审批时不会超卖名额
- 重复申请、重复加入、重复退出均被正确拦截
- 组局状态流转与成员数据一致

---

## 5.3 聊天与消息域

### 目标

支撑组局聊天室与用户私聊的稳定消息链路。

### 涉及表

- `messages`
- `direct_messages`
- `notifications`
- `banned_words`

### 开发项

1. **消息发送接口收口**
   - 文本消息优先
   - 语音/定位消息作为后续扩展

2. **消息内容校验**
   - 长度限制
   - 敏感词过滤/拦截
   - 空消息拦截

3. **发送权限校验**
   - 仅群成员可发送群聊消息
   - 仅好友/允许私聊关系可发送私聊消息（按产品规则落地）

4. **通知联动**
   - 新消息提醒
   - 申请审批结果通知
   - 系统通知

5. **未读状态与已读回执**
   - 若当前表结构尚未完整支持，可列为后续增强项

### 验收标准

- 非组成员无法向组聊发消息
- 敏感内容可被拦截或标记
- 消息写入失败时不会静默成功

---

## 5.4 通知域

### 目标

将业务副作用统一沉淀为通知中心，避免逻辑散落在前端。

### 涉及表

- `notifications`

### 通知来源

- 组局申请提交
- 申请通过/拒绝
- 被邀请入组
- 新聊天消息
- 举报处理结果
- 系统公告

### 建议

- 通知类型使用枚举统一管理
- 支持已读/未读状态
- 业务接口只产生命令，通知写入由服务端统一执行
- 后续可扩展站内信、短信、推送等多通道

---

## 5.5 举报与审核域

### 目标

建立平台治理闭环，支持用户举报、后台审核、处罚执行。

### 涉及表

- `reports`
- `reviews`
- `profiles`
- `credit_history`
- `notifications`

### 开发项

1. **举报提交接口**
   - 举报对象类型（用户/消息/组局）
   - 举报原因分类
   - 补充说明
   - 证据材料（如后续支持）

2. **审核处理接口**
   - 通过/驳回
   - 处罚类型：警告、扣分、禁言、封禁、限制创建/加入

3. **处罚落地**
   - 更新用户状态
   - 写入信用分历史
   - 发送处理通知

4. **审计要求**
   - 审核人
   - 审核时间
   - 审核理由
   - 处罚结果

### 验收标准

- 举报数据可追踪
- 审核结果有审计记录
- 处罚动作与用户状态一致

## 6. 数据与约束设计建议

## 6.1 必补约束

建议逐表核查并补齐以下约束：

1. `group_members`
   - 唯一约束：`(group_id, user_id)`

2. `join_requests`
   - 对“待处理申请”增加唯一限制，避免同一用户对同一组局重复提交待审核申请

3. `friendships`
   - 用户对关系去重，避免双向重复记录

4. `notifications`
   - 对关键业务通知增加业务幂等键（如 `biz_type + biz_id + receiver_id`）

5. `credit_history`
   - 每次信用分变更必须有来源类型、来源 ID、变更值、操作者

## 6.2 索引建议

优先补充或核查以下索引：

- `groups(city_id, start_time, status)`
- `groups(host_id, created_at desc)`
- `group_members(group_id)`
- `group_members(user_id)`
- `join_requests(group_id, status)`
- `join_requests(user_id, created_at desc)`
- `messages(group_id, created_at)`
- `direct_messages(conversation_key, created_at)` 或等价会话索引
- `notifications(user_id, is_read, created_at desc)`
- `reports(status, created_at desc)`

## 6.3 枚举与字典统一

建议统一梳理并固化以下枚举：

- `group_status`
- `join_request_status`
- `notification_type`
- `report_status`
- `review_result`
- `message_type`

目标：

- 前后端统一语义
- 避免魔法字符串散落
- 降低联调歧义

## 7. 服务端接口/RPC 规划

建议优先建设如下受控接口（名称可按实际规范调整）：

| 能力 | 建议形态 | 说明 |
| --- | --- | --- |
| 创建组局 | RPC / Edge Function | 统一校验资格、时间、限额、初始成员 |
| 申请加入组局 | RPC / Edge Function | 统一处理重复申请、资格校验、通知 |
| 审批加入申请 | RPC | 强事务，防止超卖 |
| 退出组局 | RPC | 统一处理成员关系、状态、通知、信用分 |
| 发送群聊消息 | Edge Function | 统一做成员校验、敏感词校验 |
| 发送私聊消息 | Edge Function | 统一做关系校验、频控、敏感词校验 |
| 提交举报 | Edge Function | 统一做参数校验与通知 |
| 审核举报 | Edge Function / 后台接口 | 仅管理员可调用 |
| 标记通知已读 | RPC / 普通更新 | 简单状态流转 |

### 接口设计原则

- 请求参数显式，不依赖前端隐式约定
- 错误码可枚举，不直接暴露底层 SQL 错误给前端
- 对关键接口增加 `request_id` / 幂等键支持
- 返回结果区分“业务失败”和“系统失败”

## 8. 风控与安全计划

## 8.1 权限控制

至少区分：

- 未登录用户
- 普通用户
- 组局主理人
- 管理员 / 审核员

所有写操作需要同时经过：

- 认证校验
- 资源归属校验
- 业务状态校验

## 8.2 频控策略

建议先落地以下频控：

- 登录验证码发送频率
- 创建组局频率
- 申请加入频率
- 私聊消息发送频率
- 举报提交频率

可基于：

- 用户 ID
- IP
- 设备标识（后续）
- 时间窗口

## 8.3 敏感词与内容治理

- `banned_words` 维护敏感词库
- 消息与昵称创建时统一校验
- 支持“拦截发送”或“替换+标记审核”两种策略

## 8.4 审计与追踪

关键动作必须可追踪：

- 谁在什么时间创建了组局
- 谁审批了谁的加入申请
- 谁审核了哪条举报
- 谁触发了封禁/扣分

建议后续补充：

- 审计日志表
- 错误日志采集
- 异常告警通知

## 9. 任务与自动化计划

当前需求说明中已经提到性能测试与优化待完善，结合业务特点，建议引入以下任务：

### 9.1 定时任务

1. **组局状态自动流转**
   - 未开始 → 进行中 → 已结束

2. **过期申请自动关闭**
   - 长时间未处理的 `join_requests` 自动转为过期/关闭

3. **过期通知归档/清理**
   - 降低列表查询压力

4. **验证码清理**
   - 清理过期 OTP

5. **风控扫描任务**
   - 扫描异常高频用户与可疑行为

### 9.2 数据脚本

建议后续提供 Python/Node.js 脚本用于：

- 历史数据修复
- 批量补通知
- 信用分重算
- 数据导出与统计
- 运营巡检

## 10. 测试与验收计划

## 10.1 自测策略

后端开发遵循“关键规则先测后改”的原则，重点覆盖：

### 单元/函数级验证

- 参数校验
- 状态流转
- 权限校验
- 枚举合法性
- 敏感词匹配

### 集成验证

- 创建组局 → 申请加入 → 审批通过 → 通知生成
- 创建组局 → 申请加入 → 拒绝 → 通知生成
- 群聊消息发送 → 敏感词拦截
- 举报提交 → 审核通过 → 用户处罚生效

### 并发验证

重点验证：

- 最后一个名额被并发审批
- 重复点击提交申请
- 重复退出组局
- 重复发送通知

## 10.2 验收基线

建议以以下标准作为本阶段验收口径：

1. 关键写操作已从前端散落逻辑收口到服务端
2. 核心表具备必要唯一约束与索引
3. 关键业务失败可返回明确错误信息
4. 并发下不会出现明显数据错乱
5. 关键治理链路（举报/审核/处罚）可闭环
6. 基础性能满足需求说明中的优化目标推进前提

## 11. 开发排期建议

目标：先守住最容易出数据问题的主流程。

- 创建组局接口

- 审批加入申请 RPC

## 12. 风险项与依赖项

## 12.1 风险项

1. **现有 RLS / 约束未知**
   - 需要尽快核查当前 Supabase migration、policy、function 实际状态

2. **前端已直接依赖表结构**
   - 后端收口时可能引发前端联调改造成本

3. **历史数据可能不满足新约束**
   - 在补唯一约束前需要先做脏数据清理

4. **聊天与通知可能涉及实时能力**
   - 需要评估 Supabase Realtime 的成本与一致性边界

5. **治理规则尚需产品进一步细化**
   - 如主理人退出规则、活动开始后退出规则、信用分扣减标准

## 12.2 外部依赖

需要与产品/前端/技术 Leader 进一步对齐：

- 加入组局是否必须审批
- 主理人退出后的组局归属规则
- 私聊权限规则
- 举报分类与处罚标准
- 通知类型与文案模板
- 性能压测场景与指标口径

---

## 13. 下一步执行建议

建议按以下顺序落地：

1. **盘点 Supabase 现状**
   - 核查 migration、RLS policy、trigger、function 是否已存在
2. **确定第一批服务端收口接口清单**
   - 优先锁定组局主流程 4 个接口
3. **补数据约束与索引**
   - 先清理历史脏数据，再加唯一约束
4. **与前端对齐接口契约**
   - 明确字段、错误码、联调顺序
5. **补自测用例与并发验证脚本**
   - 重点覆盖最后名额、重复申请、重复退出
6. **逐步将聊天、通知、举报纳入服务端控制**

## 14. 结论

`my_que_you` 当前已经具备较完整的业务数据模型，但后端能力仍处于“BaaS 数据底座可用、服务端规则仍需收口”的阶段。接下来的重点不是盲目扩库扩表，而是：

- 先把关键业务规则收回服务端
- 再补齐风控、治理、任务与可观测性
- 最后再推进独立服务化演进

这样可以在不推翻现有架构的前提下，逐步建立一个正确、稳定、可维护的后端体系，为后续用户增长、联调提效和平台治理打下基础。
