# 雀友聚产品路线图

## 1. 路线图目标

围绕“更快成局、更高到场、更低纠纷”三个核心目标，分阶段建设雀友聚的供给、匹配、履约与运营能力。

## 2. 规划原则

- **先闭环后增长**：优先保证用户能完成找局、建局、入局、沟通、履约完整链路。
- **先信任后扩张**：先补齐信用、举报、评价等基础约束，再放大规模。
- **先重点城市后复制**：以核心城市验证模型，再复制到更多城市。

## 3. 版本路线图

| 阶段 | 时间建议 | 主题 | 核心交付 | 成功标准 | 主要仓库 |
| ---- | -------- | ---- | -------- | -------- | -------- |
| Phase 1 / V1.0 | 2~4 周 | 核心闭环上线 | 登录注册、城市/LBS、拼团创建、申请审核、群聊、基础通知 | 用户可完成建局与入局闭环；成局率达到基础目标 | my_que_you |
| Phase 2 / V1.1 | 4~8 周 | 转化与履约优化 | 推荐排序、邀请裂变、评价增强、开局提醒、取消补位机制 | 首次申请率、到场率明显提升 | my_que_you + que_you_app |
| Phase 3 / V1.2 | 8~12 周 | 运营与增长 | banner/活动位、城市运营工具、用户分层触达、数据看板 | 支撑城市冷启动和精细化运营 | my_que_you |
| Phase 4 / V2.0 | 12 周+ | 社区沉淀与商业化 | 俱乐部、长期局、会员权益、增值曝光、更多桌游场景 | 留存提升并验证收入模型 | my_que_you + que_you_app |

## 4. 按仓库拆解

### 4.1 my_que_you

**定位：** Web 管理与业务主站能力承载。

#### P0
- 用户注册登录与基础资料
- 城市选择、LBS 列表、拼团详情
- 创建拼团、申请加入、房主审核
- 群聊与系统消息
- 举报处理、基础后台配置

#### P1
- 推荐排序与筛选优化
- 信用分、标签化评价
- 运营 banner、活动位、消息触达
- 数据看板与核心指标追踪

#### P2
- 俱乐部/固定牌友圈
- 会员体系与增值曝光
- 多品类局扩展

### 4.2 que_you_app

**定位：** 移动端深度体验与高频触达承载。

#### P1
- 更轻量的找局/入局路径
- 开局提醒、审核结果、群消息推送
- 分享拉新与好友邀请

#### P2
- 移动端履约签到/到场确认
- 实时位置辅助到店
- 个人信用档案与历史局记录沉淀

## 5. 关键里程碑

| 里程碑 | 目标 | 交付物 | 风险检查点 |
| ------ | ---- | ------ | ---------- |
| M1：需求冻结 | 明确 V1.0 范围 | SRS、路线图、验收标准 | 是否仍有范围漂移 |
| M2：开发完成 | 核心闭环功能完成联调 | 可提测版本 | 聊天、审核、通知链路是否稳定 |
| M3：灰度上线 | 在核心城市小范围开放 | 灰度反馈清单 | 供给不足、举报问题是否可控 |
| M4：正式上线 | 对目标城市开放 | 上线复盘与数据日报 | 成局率、到场率是否达标 |

## 6. 风险提醒

- 若核心城市供给不足，路线图需优先增加运营补贴与房主激励，而非继续扩功能。
- 若举报率和爽约率高于预期，应提前插入信用与仲裁能力，压缩增长功能排期。
- 若移动端消息触达不足，que_you_app 的推送能力优先级需要上调。

## 7. 对老板的阶段同步口径

- **当前目标：** 先把 V1.0 核心闭环做稳，确保用户能真实完成一场局。
- **当前进展：** 已补齐需求规格与路线图，便于研发按阶段执行。
- **当前风险：** 冷启动供给、履约信任、通知触达是上线前三大风险。
- **下一步：** 由技术 Leader 评审实现范围，再同步详细里程碑与资源排期。
