# 雀友聚 V2 任务执行计划

## 1. 文档目的

本计划用于承接当前已确认必须在 V2 实现的需求，并将《需求文档》中确定的版本范围同步拆解为可执行任务包，为排期、研发评估、测试准备和上线验收提供依据。

---

## 2. 与需求文档对齐说明

当前任务执行计划已与《memory-bank/v2/需求文档.md》保持一致，V2 仅包含以下四类能力：

1. 账号与合规能力补齐；
2. 拼团效率与履约增强；
3. 社交安全与信任增强；
4. 增长基础能力。

不属于上述范围、且已在需求文档中明确移出的能力，不再纳入本计划，统一进入后续版本规划。

## 4. 任务包拆解

### T9 账号合规与身份增强

**对应需求文档章节**：3.1 账号与合规能力补齐  
**目标**：补齐账号生命周期管理与用户身份认证能力，满足平台合规要求，并增强身份可信度。

#### 4.1.1 账号注销完整闭环（P0）

**目标说明**：支持用户发起注销申请、经过冷静期后完成账号状态流转、数据处理和结果通知，形成可执行、可追踪的注销流程。

**任务拆解：**

1. 前台用户侧
   - 提供账号注销申请入口；
   - 展示注销前须知、冷静期说明与风险提示；
   - 展示注销申请结果、处理中状态与最终完成状态。
2. 后台与状态流转
   - 建立注销申请记录；
   - 支持冷静期状态管理；
   - 支持撤销申请或冷静期到期自动处理；
   - 完成账号状态变更。
3. 数据处理与通知
   - 明确用户数据保留、脱敏、删除或冻结策略；
   - 在关键节点发送通知（申请成功、撤销成功、注销完成等）。
4. 验收口径
   - 注销流程状态闭环完整；
   - 异常场景有提示与兜底；
   - 不影响 V1 已有登录、拼团、消息等核心链路。

#### 4.1.2 实名认证（P1）

**目标说明**：建立实名认证流程、认证状态展示与风控校验能力，提升平台信任度并降低虚假身份风险。

**任务拆解：**

1. 前台用户侧
   - 提供实名认证入口；
   - 支持认证信息填写/提交；
   - 展示未认证、认证中、认证通过、认证失败等状态。
2. 后台与校验流程
   - 建立认证信息存储与审核/校验流程；
   - 支持审核结果回写与失败原因展示；
   - 支持与风控规则联动。
3. 展示与使用约束
   - 在用户资料或关键场景展示认证状态；
   - 明确哪些核心功能可对未实名用户进行限制或提示。
4. 验收口径
   - 认证流程可独立跑通；
   - 状态回写准确；
   - 与用户身份、风控及关键业务场景口径一致。

**当前进展**：

- 前台入口、实名认证信息填写/提交、未认证/审核中/已实名/认证失败等状态展示已落地；
- 设置页已支持提交认证、撤销申请、失败原因展示与状态兼容处理；
- 关键场景已补充实名认证限制提示与前台阻断：
  - 创建拼团 `GroupCreate`
  - 申请加入 `GroupDetail`
- 已补齐实名认证后端能力：
  - Supabase migration：请求表、审计表、`profiles` 状态摘要字段、RLS、通知枚举
  - 用户侧 Edge Function：状态查询、提交认证、撤销申请
  - 管理侧 Edge Function：审核通过、审核驳回
  - 混合审核规则：低风险自动通过；风险用户进入人工审核；非法证件号直接驳回
- 已补齐实名认证工具函数、后端契约测试与页面集成测试，并完成本地构建验证；
- 当前仍需在真实环境继续验证：
  - 迁移已成功部署到目标 Supabase 环境
  - Edge Function 鉴权与线上路由可用
  - 服务端强制拦截、RLS 防绕过、安全与敏感数据合规口径满足预期

**状态**：✅ 前后端能力已补齐；🟡 真实环境联调 / 安全合规验收待完成

### T10 拼团效率与履约增强

**对应需求文档章节**：3.2 拼团效率与履约增强  
**目标**：提升用户申请转化、临近开局场景下的成局率和到场率。

#### 4.2.1 紧急补位（P1）

**目标说明**：当拼团临时缺人时，支持快速补位与优先曝光，减少临近开局流失。

**任务拆解：**

1. 场次状态识别
   - 支持缺人状态标记；
   - 定义哪些场景可触发“紧急补位”。
2. 前台曝光与入口
   - 提供紧急补位入口或标识；
   - 在列表、详情页或消息中提升曝光。
3. 通知与状态更新
   - 对潜在合适用户触发补位通知或推荐；
   - 补位成功后自动更新场次状态与名额信息。
4. 验收口径
   - 紧急补位场次可被识别、曝光、补位成功并完成状态回收。

**当前进展**：

- 已补齐前台紧急补位判定规则：`OPEN` 且临近开局、仍缺少少量名额的场次会被识别为“紧急补位”；
- 已在拼团列表页提升紧急补位场次曝光，展示专属标识、倒计时文案与“立即补位”入口；
- 已在拼团详情页补充紧急补位横幅提示，明确当前场次处于临近开局缺人状态；
- 已在成员退出 / 房主踢人后的状态回收中接入紧急补位提示，缺口放大后仍会自动保持开放并触发优先曝光提示；
- 已补齐紧急补位规则单测、列表页集成测试、详情页集成测试，并完成本地自动化验证。

**状态**：✅ 已完成（本地自动化通过）

#### 4.2.2 推荐排序（P1）

**目标说明**：基于距离、时间、偏好等信息优化列表排序，提升拼团点击率与申请转化。

**任务拆解：**

1. 排序策略定义
   - 明确基础排序规则与权重因子；
   - 支持距离、时间、偏好等要素接入。
2. 列表接入与效果验证
   - 在首页/找局列表中落地新的排序策略；
   - 对比排序前后点击率、申请率等效果。
3. 兜底与兼容
   - 排序条件缺失时提供默认策略；
   - 确保不影响基础筛选、搜索与列表稳定性。
4. 验收口径
   - 列表排序逻辑可解释、可验证，效果指标具备追踪口径。

**当前进展**：

- 已抽离共享推荐排序能力：基于 `距离 + 开局时间 + 偏好玩法 + 可加入状态` 生成可解释分数，并保留“紧急补位优先”规则；
- 已改为从用户历史参与 / 主持过的牌局中推断玩法偏好，避免当前 `OPEN` 实时列表反向污染推荐结果；
- 已在首页 `Home` 与找局列表 `Index` 接入同一套推荐排序逻辑，默认排序切换为 `智能推荐`，同时保留距离、最近开始、最新发布等手动排序兜底；
- 首页已补充默认定位请求，确保智能推荐在首页场景下也能真实接入距离因子；
- 已补齐推荐排序规则单测、首页 / 列表页集成测试，覆盖距离优先、时间优先、玩法偏好加权、缺失条件兜底、开放场次优先、首页定位请求等关键场景；
- 已完成本地自动化验证与前端构建验证，真实点击率 / 申请率效果对比待 V2 联调阶段继续补证。

**状态**：✅ 已完成（本地自动化通过；真实效果指标待 V2 联调统一验证）

#### 4.2.3 开局提醒（P1）

**目标说明**：在开局前对房主和成员进行提醒，提升到场率，降低爽约。

**任务拆解：**

1. 提醒策略
   - 明确提醒触发时间点；
   - 区分房主与成员提醒内容。
2. 触达方式
   - 站内消息、订阅消息或其他提醒方式接入；
   - 记录提醒发送结果。
3. 异常与补充机制
   - 开局时间变更时更新提醒策略；
   - 已取消场次不再重复提醒。
4. 验收口径
   - 提醒发送准确、对象正确、状态可追踪。

**当前进展**：

- 已明确开局提醒策略：默认在开局前 `30 分钟` 对房主与成员分别发送一次站内提醒；
- 已补齐提醒计划共享规则：支持房主/成员角色去重、提醒时间计算、已取消场次拦截与开局时间变更导致的旧提醒作废判定；
- 已新增提醒追踪落库能力：
  - Supabase migration：新增 `group_start_reminders` 表、提醒状态字段、唯一索引、RLS 与更新时间触发器
  - 通知类型：新增 `group_start_reminder`
- 已新增定时发送 Edge Function：
  - `auto-send-group-start-reminders`
  - 基于当前 `groups.start_time` 和成员列表动态生成提醒对象
  - 发送后回写 `sent / failed / cancelled / superseded` 状态，支持时间变更与取消场次兜底
- 已在消息中心补充开局提醒图标映射，并完成基础展示/点击链路回归；
- 已补齐开局提醒规则单测、发送编排契约测试，并完成 T10 范围本地自动化回归与前端构建验证；
- 当前仍需在真实环境继续验证：
  - migration 已成功部署到目标 Supabase 环境；
  - `auto-send-group-start-reminders` 定时任务已接入真实调度；
  - `verify_jwt = false` 的调度入口具备预期的触发来源限制与调用保护；
  - 真实通知落库、多账号收件、时间变更重排与取消场次拦截符合预期。

**状态**：✅ 已完成（本地自动化通过；真实调度 / 真实通知落库 / 多账号联调待 V2 统一执行）

### T11 社交安全与信任增强

**对应需求文档章节**：3.3 社交安全与信任增强  
**目标**：增强用户社交安全感与优质身份识别能力，提升匹配可信度。

#### 4.3.1 黑名单屏蔽（P1）

**目标说明**：支持用户拉黑、屏蔽互动与消息隔离，提升用户安全感与社交可控性。

**任务拆解：**

1. 用户操作能力
   - 支持用户将他人加入黑名单；
   - 支持取消拉黑。
2. 互动隔离规则
   - 被拉黑后限制私聊、互动、推荐曝光或其他关键接触场景；
   - 明确双向/单向屏蔽规则。
3. 管理能力
   - 提供黑名单管理页；
   - 支持查看、移除黑名单成员。
4. 验收口径
   - 拉黑后关键互动链路隔离生效，取消拉黑后可恢复。

#### 4.3.2 标签化评价与履约档案增强（P1）

**目标说明**：在基础评价外增加标签化评价与历史履约档案沉淀，提高匹配质量并降低纠纷。

**任务拆解：**

1. 评价结构增强
   - 设计标签化评价项；
   - 保留基础评价信息并扩展结构化标签。
2. 履约档案沉淀
   - 建立历史履约记录汇总逻辑；
   - 支持用户查看关键履约摘要。
3. 展示与使用场景
   - 在用户主页、申请审核或匹配决策场景中展示；
   - 明确展示边界，避免信息过载。
4. 验收口径
   - 评价数据可沉淀、可展示、可用于辅助匹配判断。

**状态**：⏳ 待排期

### T12 裂变增长基础建设

**对应需求文档章节**：3.4 增长基础能力  
**目标**：提升拼团传播效率与邀请拉新可追踪能力，支撑基础增长动作。

#### 4.4.1 分享海报（P1）

**目标说明**：支持拼团分享海报生成与传播，提升裂变传播效率。

**任务拆解：**

1. 海报内容生成
   - 明确海报信息结构，如场次、时间、地点、人数等；
   - 生成统一视觉模板。
2. 分享入口
   - 在拼团详情或关键转化场景提供海报生成/分享入口；
   - 支持用户保存或转发。
3. 传播校验
   - 校验海报信息准确性；
   - 校验分享后回流链路是否正常。
4. 验收口径
   - 海报可成功生成、展示信息完整、传播链路可用。

#### 4.4.2 邀请码（P1）

**目标说明**：支持邀请码机制与邀请归因，降低拉新成本并支撑渠道归因。

**任务拆解：**

1. 邀请码生成与绑定
   - 支持用户生成或绑定邀请码；
   - 明确邀请码使用时机。
2. 邀请关系记录
   - 建立邀请人、被邀请人关系记录；
   - 记录绑定时间与生效状态。
3. 归因与查询
   - 定义基础归因口径；
   - 支持后台查询、统计或导出。
4. 验收口径
   - 邀请关系可记录、可查询、可用于基础增长统计。

**状态**：⏳ 待排期

## 6. 执行原则

1. V2 仅围绕《需求文档》已明确收敛的 P0 / P1 范围推进，不新增中长期能力；
2. 每个任务包都需至少覆盖前台入口、后台治理、状态流转、通知链路和验收口径；
3. 若新增需求不直接服务于合规、安全、成局效率或增长基础，默认进入后续计划；
4. 正式进入研发前，需补齐 PRD、接口影响、测试范围、灰度方案与回滚预案；
5. 若后续 V2 需求文档发生变更，任务执行计划需同步更新，保持一一对应关系。
