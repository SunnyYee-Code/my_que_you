## 2. 验收口径

### 2.1 已具备上线基础

以下情况可认定为“已具备上线基础”：

- 需求对应的前台入口、状态反馈、数据结构、后端能力基本齐备；
- 关键业务链路已形成闭环；
- 已有对应自动化测试，且当前仓库实际执行通过；
- 不依赖额外真实环境能力即可完成主要验收。

### 2.2 待真实环境验收

以下情况认定为“待真实环境验收”：

- 本地代码与自动化测试已基本齐备；
- 但仍依赖真实 Supabase、真实账号、多账号联动、RLS、Edge Function 鉴权、定时调度、真实通知、真实设备分享等外部条件；
- 当前只能认定为“开发完成度较高”，不能直接视为最终上线验收完成。

### 2.3 需修复后再验收

以下情况认定为“需修复后再验收”：

- 已发现真实链路阻断或明显缺口；
- 自动化测试覆盖了前端或局部逻辑，但未覆盖到真实问题点；
- 若不修复，需求在真实环境中大概率无法正常闭环。

### 2.4 当前验收依据

本轮验收主要基于以下证据：

从代码实现角度看，V2 的 9 个需求项已基本全部进入“有代码落点、有数据结构、有测试基础”的阶段，**整体开发完成度较高**。

从功能闭环角度看，前期发现的 3 个高优先级代码问题已完成修复：账号注销鉴权已补齐、快速加入实名绕过已封堵、开局提醒调度入口保护与 cron 注册已补齐。但 V2 仍普遍缺少真实环境联调证据，因此当前更准确的结论是：

- **0 项处于“代码阻断未修复”状态**
- **9 项开发已基本完成，但仍待真实环境 / 真实设备 / 多账号场景验收**

从测试覆盖角度看，V2 的单元测试和集成测试基础已经明显优于“仅代码落地”的状态，但**尚未形成 V2 专项 E2E 自动化验收体系**。因此，当前可以认定 V2 已进入“准验收状态”，但还不能直接认定为“全部正式验收完成”。
