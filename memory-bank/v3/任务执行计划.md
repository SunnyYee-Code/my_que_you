# 雀友聚 V3 任务执行计划

## 1. 文档目的

本计划用于承接当前已确认进入 V3 的产品需求，并将《memory-bank/v3/需求文档.md》中已冻结的版本范围拆解为可执行任务包，为排期评估、研发实施、测试准备和上线验收提供依据。

---

## 2. 与需求文档对齐说明

当前任务执行计划已与《memory-bank/v3/需求文档.md》保持一致，V3 仅包含以下四类能力：

1. 运营效率与履约体验增强；
2. 社区关系沉淀与用户留存增强；
3. 沟通与线下到场协同增强；
4. 商业化与品类扩展探索。

本计划遵循以下对齐原则：

1. 不把“尚未验证”直接视为“尚未实现”，V2 遗留能力需先做实现盘点，再确定实际补齐范围；
2. 不把联调、灰度、验收本身写成产品需求，但会在任务执行计划中体现为交付动作；
3. 不把 V1 已经稳定上线的能力重新定义为 V3 新需求；
4. 不新增《需求文档》之外的版本内容，所有任务包都必须能回指到 V3 需求条目。

## 4. 任务包拆解

### T14 运营效率与履约体验增强

**对应需求文档章节**：4.2 运营效率与履约体验增强
**目标**：通过降低发团成本、提升消息触达、增强运营配置能力和补齐履约数据，提升日常运营效率与用户到场体验。

#### 4.2.1 常用地点收藏（P1）

**目标说明**：支持房主收藏高频开局地点，并在创建拼团时快捷复用，减少重复填写成本。
**任务拆解：**

1. 收藏能力
   - 支持用户新增、编辑、删除常用地点；
   - 保存地点名称、地址、坐标和备注等必要信息。
2. 创建链路接入
   - 在创建拼团时提供常用地点快捷选择入口；
   - 支持选中后自动回填地点信息。
3. 管理与约束
   - 设计数量上限、排序规则和最近使用优先策略；
   - 处理失效地点、跨城市切换和定位缺失场景。
4. 验收口径
   - 常用地点可被维护、可被复用；
   - 创建拼团效率有明确提升，且不影响原有定位和手动输入链路。

#### 4.2.2 消息触达增强（P1）

**目标说明**：对审核结果、开局提醒、补位提醒、举报处理等关键状态增强触达方式与召回策略，提高关键动作到达率和用户回流率。
**状态**：✅ 已完成

**任务拆解：**

1. 触达分层
   - 梳理高优先级状态事件，定义站内、订阅、短信或其他触达层级；
   - 明确各类事件的发送时机与优先级。
2. 召回策略
   - 为未读或未响应场景设计二次召回规则；
   - 区分房主、成员和被处理用户的消息内容。
3. 发送与追踪
   - 记录发送状态、送达结果和用户回流行为；
   - 支持失败兜底和频控规则。
4. 验收口径
   - 关键状态触达准确、不过度打扰；
   - 消息结果可追踪，可用于后续运营复盘。

#### 4.2.3 banner / 活动位（P1）

**目标说明**：支持首页或社区页配置活动位、主题活动和冷启动投放，提高运营活动承接能力。
**状态**：✅ 已完成

**任务拆解：**

1. 运营配置能力
   - 支持创建、编辑、上下线 banner / 活动位配置；
   - 支持标题、图片、跳转链接、展示时间和城市范围等字段。
2. 前台展示能力
   - 在首页或社区页接入活动位展示；
   - 支持多活动排序、曝光控制和点击跳转。
3. 统计与兜底
   - 记录曝光、点击和转化基础数据；
   - 处理空配置、过期配置和异常跳转场景。
4. 验收口径
   - 活动位可由运营独立配置和发布；
   - 展示与点击链路稳定，数据可回收。

#### 4.2.4 数据看板（P1）

**目标说明**：提供注册、申请、成局、到场、举报、分享、邀请归因等关键指标看板，支撑运营与管理决策。
**状态**：✅ 已完成

**任务拆解：**

1. 指标口径定义
   - 冻结注册、申请、成局、到场、举报、分享、邀请等核心指标；
   - 明确时间维度、城市维度和渠道维度。
2. 数据汇总与展示
   - 建立看板查询能力和基础可视化展示；
   - 支持关键趋势、环比和分布查看。
3. 管理侧使用
   - 在管理后台接入数据看板入口；
   - 支持按时间和城市筛选。
4. 验收口径
   - 指标可对账、可解释；
   - 看板能够支撑日常运营复盘和版本评估。

#### 4.2.5 履约签到 / 到场确认（P1）

**目标说明**：增加签到、到场确认与爽约记录沉淀能力，提高真实履约判断准确度。
**状态**：✅ 已完成

**任务拆解：**

1. 到场确认机制
   - 支持房主或成员在开局前后发起签到 / 到场确认；
   - 明确签到时间窗和确认规则。
2. 爽约记录沉淀
   - 对未签到、被确认未到场或异常离场场景沉淀履约记录；
   - 与履约档案能力打通。
3. 展示与申诉
   - 在用户资料、审核决策或管理侧展示履约记录摘要；
   - 预留异常纠偏或申诉处理口径。
4. 验收口径
   - 到场记录可沉淀、可查询、可用于后续信用判断；
   - 不制造明显误判或滥用空间。

#### 4.2.6 性能优化（P1）

**目标说明**：持续优化核心链路体验与系统稳定性，支撑城市放量和更高频的运营动作。

**状态**：✅ 已完成

**任务拆解：**

1. 性能基线
   - 梳理首页、列表、详情、聊天、创建、申请等核心链路性能指标；
   - 建立可对比的基线数据。
2. 优化治理
   - 优先处理慢查询、重复请求、列表渲染和资源加载等问题；
   - 清理影响稳定性的构建告警和高风险技术债。
3. 监控与回归
   - 增加关键性能监控和异常告警；
   - 建立版本回归和容量验证机制。
4. 验收口径
   - 核心链路性能指标有明确改善；
   - 稳定性风险可监控、可回溯、可预警。
