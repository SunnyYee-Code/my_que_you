# 雀友聚 V3 任务执行计划

## 1. 文档目的

本计划用于承接当前已确认进入 V3 的产品需求，并将《memory-bank/v3/需求文档.md》中已冻结的版本范围拆解为可执行任务包，为排期评估、研发实施、测试准备和上线验收提供依据。

---

## 2. 与需求文档对齐说明

当前任务执行计划已与《memory-bank/v3/需求文档.md》保持一致，V3 仅包含以下四类能力：

1. 运营效率与履约体验增强；
2. 社区关系沉淀与用户留存增强；
3. 沟通与线下到场协同增强；
4. 商业化与品类扩展探索。

本计划遵循以下对齐原则：

1. 不把“尚未验证”直接视为“尚未实现”，V2 遗留能力需先做实现盘点，再确定实际补齐范围；
2. 不把联调、灰度、验收本身写成产品需求，但会在任务执行计划中体现为交付动作；
3. 不把 V1 已经稳定上线的能力重新定义为 V3 新需求；
4. 不新增《需求文档》之外的版本内容，所有任务包都必须能回指到 V3 需求条目。

## 4. 任务包拆解

### T14 运营效率与履约体验增强

**对应需求文档章节**：4.2 运营效率与履约体验增强
**目标**：通过降低发团成本、提升消息触达、增强运营配置能力和补齐履约数据，提升日常运营效率与用户到场体验。

#### 4.2.1 常用地点收藏（P1）

**目标说明**：支持房主收藏高频开局地点，并在创建拼团时快捷复用，减少重复填写成本。
**任务拆解：**

1. 收藏能力
   - 支持用户新增、编辑、删除常用地点；
   - 保存地点名称、地址、坐标和备注等必要信息。
2. 创建链路接入
   - 在创建拼团时提供常用地点快捷选择入口；
   - 支持选中后自动回填地点信息。
3. 管理与约束
   - 设计数量上限、排序规则和最近使用优先策略；
   - 处理失效地点、跨城市切换和定位缺失场景。
4. 验收口径
   - 常用地点可被维护、可被复用；
   - 创建拼团效率有明确提升，且不影响原有定位和手动输入链路。

#### 4.2.2 消息触达增强（P1）

**目标说明**：对审核结果、开局提醒、补位提醒、举报处理等关键状态增强触达方式与召回策略，提高关键动作到达率和用户回流率。
**状态**：✅ 已完成

**任务拆解：**

1. 触达分层
   - 梳理高优先级状态事件，定义站内、订阅、短信或其他触达层级；
   - 明确各类事件的发送时机与优先级。
2. 召回策略
   - 为未读或未响应场景设计二次召回规则；
   - 区分房主、成员和被处理用户的消息内容。
3. 发送与追踪
   - 记录发送状态、送达结果和用户回流行为；
   - 支持失败兜底和频控规则。
4. 验收口径
   - 关键状态触达准确、不过度打扰；
   - 消息结果可追踪，可用于后续运营复盘。

#### 4.2.3 banner / 活动位（P1）

**目标说明**：支持首页或社区页配置活动位、主题活动和冷启动投放，提高运营活动承接能力。

**任务拆解：**

1. 运营配置能力
   - 支持创建、编辑、上下线 banner / 活动位配置；
   - 支持标题、图片、跳转链接、展示时间和城市范围等字段。
2. 前台展示能力
   - 在首页或社区页接入活动位展示；
   - 支持多活动排序、曝光控制和点击跳转。
3. 统计与兜底
   - 记录曝光、点击和转化基础数据；
   - 处理空配置、过期配置和异常跳转场景。
4. 验收口径
   - 活动位可由运营独立配置和发布；
   - 展示与点击链路稳定，数据可回收。

#### 4.2.4 数据看板（P1）

**目标说明**：提供注册、申请、成局、到场、举报、分享、邀请归因等关键指标看板，支撑运营与管理决策。

**任务拆解：**

1. 指标口径定义
   - 冻结注册、申请、成局、到场、举报、分享、邀请等核心指标；
   - 明确时间维度、城市维度和渠道维度。
2. 数据汇总与展示
   - 建立看板查询能力和基础可视化展示；
   - 支持关键趋势、环比和分布查看。
3. 管理侧使用
   - 在管理后台接入数据看板入口；
   - 支持按时间和城市筛选。
4. 验收口径
   - 指标可对账、可解释；
   - 看板能够支撑日常运营复盘和版本评估。

#### 4.2.5 履约签到 / 到场确认（P1）

**目标说明**：增加签到、到场确认与爽约记录沉淀能力，提高真实履约判断准确度。

**任务拆解：**

1. 到场确认机制
   - 支持房主或成员在开局前后发起签到 / 到场确认；
   - 明确签到时间窗和确认规则。
2. 爽约记录沉淀
   - 对未签到、被确认未到场或异常离场场景沉淀履约记录；
   - 与履约档案能力打通。
3. 展示与申诉
   - 在用户资料、审核决策或管理侧展示履约记录摘要；
   - 预留异常纠偏或申诉处理口径。
4. 验收口径
   - 到场记录可沉淀、可查询、可用于后续信用判断；
   - 不制造明显误判或滥用空间。

#### 4.2.6 性能优化（P1）

**目标说明**：持续优化核心链路体验与系统稳定性，支撑城市放量和更高频的运营动作。

**任务拆解：**

1. 性能基线
   - 梳理首页、列表、详情、聊天、创建、申请等核心链路性能指标；
   - 建立可对比的基线数据。
2. 优化治理
   - 优先处理慢查询、重复请求、列表渲染和资源加载等问题；
   - 清理影响稳定性的构建告警和高风险技术债。
3. 监控与回归
   - 增加关键性能监控和异常告警；
   - 建立版本回归和容量验证机制。
4. 验收口径
   - 核心链路性能指标有明确改善；
   - 稳定性风险可监控、可回溯、可预警。

### T15 社区关系沉淀与用户留存增强

**对应需求文档章节**：4.3 社区关系沉淀与用户留存增强
**目标**：在已有拼团、好友和私信能力基础上，增强用户身份识别、关系沉淀和社区互动，提升复玩率与长期留存。

#### 4.3.1 信用勋章系统（P1）

**目标说明**：基于履约、评价和活跃度沉淀可视化勋章与身份标识，强化优质用户识别与正向激励。

**任务拆解：**

1. 勋章规则
   - 定义履约、活跃、评价等勋章获取条件；
   - 明确勋章等级、有效期和回收策略。
2. 展示场景
   - 在资料页、拼团详情、审核场景等位置展示勋章与身份标识；
   - 控制展示密度，避免信息噪音。
3. 数据联动
   - 与履约档案、签到记录、活跃数据打通；
   - 支持后续运营调整勋章规则。
4. 验收口径
   - 勋章规则可解释、可追踪；
   - 能够帮助识别优质用户并形成正向反馈。

#### 4.3.2 局后社交沉淀增强（P1）

**目标说明**：在 V1 已有好友和私信基础上，进一步沉淀复局关系、常约对象和关系标签，增强用户关系复用。

**任务拆解：**

1. 关系沉淀
   - 基于共同参与记录沉淀复局关系和常约对象；
   - 支持关系强度或互动频次统计。
2. 关系标签
   - 支持用户对常见牌友添加轻量关系标签；
   - 支持仅自己可见的管理视图。
3. 复用入口
   - 在建局、邀请、消息或个人主页中提供关系复用入口；
   - 支持快速找到近期常约对象。
4. 验收口径
   - 关系沉淀可持续更新；
   - 能真实降低复局发起成本，提升社交留存。

#### 4.3.3 雀友圈（P2）

**目标说明**：提供轻量内容发布、动态互动和约局话题沉淀能力，提升社区活跃度和内容留存。

**任务拆解：**

1. 内容发布
   - 支持图文或轻量动态发布；
   - 支持约局相关话题和基础内容审核。
2. 互动能力
   - 支持点赞、评论或轻量互动；
   - 明确互动通知与内容可见范围。
3. 信息流治理
   - 设计推荐、排序、屏蔽和举报能力；
   - 控制内容噪音和低质量刷屏。
4. 验收口径
   - 用户可稳定发布和互动；
   - 社区内容可管理、可沉淀、可回流到约局场景。

#### 4.3.4 俱乐部（P2）

**目标说明**：支持稳定圈层管理、成员沉淀和固定组织能力，提升高频用户组织效率。

**任务拆解：**

1. 俱乐部基础结构
   - 支持创建俱乐部、设置简介、成员规则和管理员角色；
   - 明确公开 / 私密俱乐部边界。
2. 成员管理
   - 支持邀请、申请加入、审核、移除和成员分层；
   - 支持俱乐部内公告或活动承接。
3. 与拼团联动
   - 支持从俱乐部内发起或关联拼团；
   - 支持俱乐部成员优先曝光或优先报名规则。
4. 验收口径
   - 俱乐部可以稳定承接固定圈层组织需求；
   - 不与现有好友、群聊和普通拼团规则冲突。

#### 4.3.5 长期局（P2）

**目标说明**：支持固定时间、固定成员或长期排期的局组织模式，承接高复玩用户需求。

**任务拆解：**

1. 排期模型
   - 支持周期性时间设置、长期成员配置和局状态管理；
   - 处理请假、补位和周期调整场景。
2. 成员协同
   - 支持固定成员确认、替补机制和长期通知策略；
   - 支持历史出勤沉淀。
3. 与现有拼团兼容
   - 明确长期局与普通拼团的关系；
   - 复用现有聊天、签到、履约和关系沉淀能力。
4. 验收口径
   - 长期局可以独立组织和维护；
   - 不破坏现有短期拼团主链路。

### T16 沟通与线下到场协同增强

**对应需求文档章节**：4.4 沟通与线下到场协同增强
**目标**：增强开局前后的沟通效率和线下集合协同能力，降低“联系不上”“找不到人”“找不到店”等到场摩擦。

#### 4.4.1 语音消息（P2）

**目标说明**：在群聊和私聊中支持语音消息发送与播放，提升高频沟通效率。

**任务拆解：**

1. 录制与发送
   - 支持语音录制、时长限制、发送和取消；
   - 处理录音权限和中断场景。
2. 播放与展示
   - 支持语音消息播放、进度展示和已读场景适配；
   - 处理弱网、资源加载失败和重复播放。
3. 内容治理
   - 设计存储策略、过期策略和举报处理口径；
   - 控制滥用和成本风险。
4. 验收口径
   - 语音消息可稳定发送和播放；
   - 不影响现有文字消息与通知链路。

#### 4.4.2 位置消息（P2）

**目标说明**：支持发送实时位置或集合点位置消息，提升线下到场协同效率。

**任务拆解：**

1. 位置发送
   - 支持发送当前位置或预设集合点位置；
   - 处理定位授权和精度异常场景。
2. 消息展示
   - 在聊天中展示位置卡片、地图预览和导航入口；
   - 支持位置失效提示。
3. 安全与隐私
   - 控制位置可见范围、有效期和撤回策略；
   - 避免位置长期暴露造成风险。
4. 验收口径
   - 位置消息可帮助用户快速会合；
   - 位置能力具备明确安全边界。

#### 4.4.3 到场辅助链路（P2）

**目标说明**：在已有导航能力基础上，进一步增加集合协同与到场辅助能力，降低线下集合摩擦。

**任务拆解：**

1. 集合协同
   - 支持发送集合提示、到店提醒和临时集合点；
   - 区分房主和成员的协同动作。
2. 到场辅助
   - 提供导航补充信息，如入口说明、楼层提示、联系人提示；
   - 支持开局前临近提醒联动。
3. 异常兜底
   - 处理改址、迟到、找不到人的异常沟通场景；
   - 与位置消息和聊天能力联动。
4. 验收口径
   - 到场前协同成本明显下降；
   - 用户能够更快完成线下集合和入场。

### T17 商业化与品类扩展探索

**对应需求文档章节**：4.5 商业化与品类扩展探索
**目标**：在不影响主链路稳定和版本聚焦的前提下，分阶段验证商业化方向与品类延展空间，为后续版本提供决策依据。

#### 4.5.1 战绩榜（P2）

**目标说明**：提供用户战绩、胜率或活跃榜单展示，增强活跃度和荣誉感。

**任务拆解：**

1. 榜单规则
   - 明确战绩、胜率、活跃等统计规则和适用前提；
   - 处理样本不足和作弊风险。
2. 展示与激励
   - 提供榜单展示入口和基础荣誉反馈；
   - 控制榜单更新频率和展示范围。
3. 风险治理
   - 避免因数据不准导致争议；
   - 明确是否先做“活跃榜”再逐步扩展到战绩榜。
4. 验收口径
   - 榜单规则可解释；
   - 能够带来正向活跃激励而非争议放大。

#### 4.5.2 记分器 / 番数计算器（P2）

**目标说明**：提供局中辅助工具，提升使用频率和工具属性。

**任务拆解：**

1. 工具定义
   - 明确记分器、番数计算器的功能边界和目标玩法；
   - 优先覆盖最常用场景。
2. 使用入口
   - 在对局前后或局中场景提供快捷入口；
   - 支持轻量独立使用。
3. 数据价值
   - 评估是否与战绩、长期局或会员权益产生联动；
   - 控制复杂度，避免反客为主。
4. 验收口径
   - 工具可独立使用、门槛低；
   - 对使用频率和停留时长有正向帮助。

#### 4.5.3 会员权益（P2）

**目标说明**：探索会员身份、权益包和权益展示体系，为商业化打基础。

**任务拆解：**

1. 权益设计
   - 明确会员可获得的身份、工具、曝光或专属服务权益；
   - 避免直接破坏社区公平性。
2. 购买与展示
   - 设计会员开通、续费、到期和权益展示链路；
   - 支持权益状态查询。
3. 风险评估
   - 评估用户接受度、转化率和投诉风险；
   - 明确首期仅做轻量验证还是正式付费闭环。
4. 验收口径
   - 会员权益定义清晰、价值明确；
   - 不伤害现有核心用户体验。

#### 4.5.4 增值曝光（P2）

**目标说明**：探索拼团置顶、优先曝光等增值服务，验证付费意愿与供需效率。

**任务拆解：**

1. 曝光模型
   - 明确置顶、优先曝光的展示规则、时效和适用场景；
   - 控制对普通排序公平性的影响。
2. 交易与履约
   - 设计购买、展示、生效和过期回收链路；
   - 支持后台审核和异常退款处理。
3. 数据验证
   - 记录曝光效果、点击率和申请转化；
   - 评估对平台整体效率的真实影响。
4. 验收口径
   - 增值曝光可被稳定配置和回收；
   - 具备清晰的效果验证和风险控制口径。

#### 4.5.5 多类型桌游扩展（P2）

**目标说明**：评估从麻将扩展到其他桌游或牌类活动的适配能力，验证品类扩张空间。

**任务拆解：**

1. 适配评估
   - 梳理当前拼团、规则、履约、聊天和评价能力中与麻将强绑定的部分；
   - 识别可抽象和不可复用的能力。
2. 小范围试点
   - 选择 1 至 2 个典型桌游类型做试点；
   - 明确活动结构、人数、规则和展示差异。
3. 版本决策依据
   - 评估用户需求、运营承接成本和技术改造成本；
   - 输出是否值得继续扩展的判断依据。
4. 验收口径
   - 形成可执行的品类扩展结论；
   - 不在 V3 内无边界扩散到完整多品类平台。
