# 雀友聚 V3 需求文档

## 1. 文档目标

本文件用于明确 `my_que_you` 在 V3 阶段需要承接的需求范围。  
V3 不再只承接“新功能规划”，而是同时覆盖两类内容：

1. **V1 / V2 已提过、但尚未形成完整上线闭环的遗留项**；
2. **真正进入 V3 的运营深化、社区沉淀、商业化与体验增强能力**。

这样处理的目的是避免版本认知失真：如果某项能力虽然已有代码落点、但仍未完成真实环境闭环、跨端协同或最终验收，就不能视为完全结束，仍需在 V3 中明确承接。

---

## 2. V3 阶段目标

V3 的目标不是继续无边界扩功能，而是在 V2 之后完成以下三件事：

1. **完成上线收口**：补齐 V1 / V2 遗留的真实环境闭环、E2E 验收和关键稳定性问题；
2. **提升留存与运营效率**：通过常用地点、消息触达、活动位、数据看板等能力提升复用率与运营效率；
3. **验证社区与商业化方向**：在不破坏主链路的前提下，逐步验证雀友圈、俱乐部、长期局及商业化扩展能力。

---

## 3. V3 版本边界说明

- **V1**：完成“找局 - 建局 - 入局 - 审核 - 沟通 - 基础治理”的最小可用闭环。
- **V2**：补齐合规、安全、履约效率和增长基础能力。
- **V3**：一方面收口 V1 / V2 尚未真正闭环的遗留项，另一方面推进运营深化、社区沉淀和商业化探索。

V3 的基本原则：

- 不把“已有代码但未闭环”的事项假设为已完成；
- 不让社区和商业化能力挤占核心主链路收口资源；
- 所有新增能力必须能明确说明对成局率、到场率、留存率或运营效率的价值。

---

## 4. V3 需求范围

### 4.1 V1 / V2 上线收口与验收补完

这部分不是“纯新功能”，但从产品交付口径上仍属于 V3 必须承接的范围。

| 需求项 | 来源 | 目标说明 | 价值 |
| --- | --- | --- | --- |
| 注册登录真实闭环补完 | V1 验收遗留 | 补齐完整注册、验证码验证、真实登录成功链路与异常兜底验收 | 确保用户准入链路真正可上线 |
| 城市定位与地图能力实景闭环 | V1 验收遗留 | 补齐真实定位授权、自动回填、地图选点与定位失败兜底 | 确保找局与建局场景可稳定使用 |
| 入局成功与成员状态同步闭环 | V1 验收遗留 | 补齐审核通过后的成员列表同步、状态联动、我的拼团记录追踪 | 防止“审核通过但用户感知未成功” |
| 实时沟通与邀请链路补完 | V1 验收遗留 | 补齐临时聊天室、实时通知、邀请好友入局的多账号真实链路验收 | 确保成局前沟通链路稳定 |
| 用户主页、信用与后台治理补完 | V1 验收遗留 | 补齐用户主页、信用限制、自动结束与信用流转、用户/拼团/聊天巡检等模块的完整验证与缺口修复 | 提升平台治理可信度 |
| 合规能力真实环境闭环 | V2 验收遗留 | 补齐账号注销、实名认证在真实 Supabase / Auth / RLS / Edge Function 环境中的最终验收 | 确保平台具备可上线的合规基础 |
| 履约提醒与状态调度闭环 | V2 验收遗留 | 补齐开局提醒 cron 调度、通知落库、时间变更重排、取消场次拦截等真实链路 | 提升到场率并降低履约风险 |
| 风险隔离与增长归因闭环 | V2 验收遗留 | 补齐黑名单、邀请码、分享海报、紧急补位、推荐排序、履约档案等真实环境验证与效果归因 | 确保 V2 新能力真正可运营、可复盘 |
| V2 专项 E2E 与回归体系 | V2 验收遗留 | 建立覆盖合规、履约、增长能力的端到端验收脚本和发布回归基线 | 降低后续迭代回归成本 |
| 核心链路性能与构建治理 | V1/V2 共性遗留 | 补齐首屏、列表、创建、申请等核心链路的性能验证、构建 warning 治理与监控口径 | 为后续放量和运营活动提供稳定基础 |

### 4.2 运营效率与履约体验增强

| 需求项 | 目标说明 | 价值 |
| --- | --- | --- |
| 常用地点收藏 | 支持房主收藏常用开局地点，创建时快捷复用 | 降低发团成本，提高重复开局效率 |
| 消息触达增强 | 对审核结果、开局提醒、补位提醒、举报处理等关键状态增强触达方式与召回策略 | 提高关键动作到达率与回流率 |
| banner / 活动位 | 支持首页/社区页配置活动位、主题活动和冷启动投放 | 提升运营投放效率与活动承接能力 |
| 数据看板 | 提供注册、申请、成局、到场、举报、分享、邀请归因等指标看板 | 让运营和管理侧可以用数据做决策 |
| 履约签到 / 到场确认 | 在移动端或关键场景增加签到、到场确认、爽约记录沉淀 | 提升履约判断准确度 |
| 性能测试与优化 | 建立压测、前端性能监控、关键接口观测和持续优化机制 | 支撑城市放量和更高频运营动作 |

### 4.3 社区关系沉淀与用户留存增强

| 需求项 | 目标说明 | 价值 |
| --- | --- | --- |
| 信用勋章系统 | 基于履约、评价、活跃度沉淀可视化勋章与身份标识 | 强化优质用户识别与正向激励 |
| 局后社交沉淀增强 | 在好友、私信基础上沉淀复局关系、常约对象和关系标签 | 提升复玩率与长期留存 |
| 雀友圈 | 提供轻量内容发布、动态互动与约局话题沉淀能力 | 提升社区活跃度和内容留存 |
| 俱乐部 / 固定牌友圈 | 支持稳定圈层管理、成员沉淀和固定组织能力 | 提升高频用户组织效率 |
| 长期局 | 支持固定时间、固定成员或长期排期的局组织模式 | 承接高复玩用户需求 |

### 4.4 沟通与线下到场协同增强

| 需求项 | 目标说明 | 价值 |
| --- | --- | --- |
| 语音消息 | 在群聊和私聊中支持语音消息发送与播放 | 提升高频沟通效率 |
| 位置消息 | 支持发送实时位置或集合点位置消息 | 提升到场协同效率 |
| 到场辅助链路 | 结合导航、位置、签到等能力增强线下集合体验 | 降低“找不到人/找不到店”的摩擦 |

### 4.5 商业化与品类扩展探索

| 需求项 | 目标说明 | 价值 |
| --- | --- | --- |
| 战绩排行榜 | 提供用户战绩、胜率或活跃榜单展示 | 增强活跃和荣誉感 |
| 记分器 / 番数计算器 | 提供局中辅助工具，提升使用频率 | 提升工具属性和使用时长 |
| 会员权益 | 探索会员身份、权益包和权益展示体系 | 为商业化打基础 |
| 增值曝光 | 探索拼团置顶、优先曝光等增值服务 | 验证付费意愿与供需效率 |
| 多类型桌游扩展 | 评估从麻将扩展到其他桌游/牌类活动的适配能力 | 验证品类扩张空间 |

## 5. V3 优先级建议

| 优先级 | 需求项 | 原因 |
| --- | --- | --- |
| P0 | V1 / V2 上线收口与验收补完 | 若不先补齐真实环境闭环、E2E 和性能基线，后续所有运营和增长投入都会建立在不稳定基础上 |
| P1 | 常用地点收藏、消息触达增强、banner / 活动位、数据看板、履约签到 / 到场确认、性能测试与优化 | 这些能力直接提升复玩率、运营效率和履约质量，且对现有主链路价值最直接 |
| P1 | 信用勋章系统、局后社交沉淀增强 | 能承接高质量用户资产沉淀，提升留存和关系复用 |
| P2 | 雀友圈、俱乐部 / 固定牌友圈、长期局 | 属于社区深化能力，价值明确，但需要建立在主链路稳定和基础数据充分之上 |
| P2 | 语音消息、位置消息、到场辅助链路 | 对体验有提升，但优先级低于基础收口和运营效率能力 |
| P2 | 战绩排行榜、记分器 / 番数计算器、会员权益、增值曝光、多类型桌游扩展 | 属于中长期活跃和商业化探索，适合在 V3 后段按资源分阶段验证 |

---

## 6. 不纳入 V3 的处理原则

以下事项默认不单独定义为独立 V3 需求项：

- 仅属于代码重构、技术栈替换但不直接产生产品结果的工作；
- 没有明确业务目标、无法说明对成局率/到场率/留存/运营效率价值的想法；
- 与当前城市验证阶段不匹配、过早投入的重资产商业化方案。

如后续出现新的强需求，统一通过产品优先级文档评估后再决定是否插入 V3。

---

## 7. V3 验收方向

1. V1 / V2 收尾项必须给出真实环境验收记录，不能只停留在“已有代码”；
2. 新增运营能力必须具备明确的数据指标与回收口径；
3. 社区与商业化能力需要先灰度验证，再决定是否全量推进；
4. 任何 V3 新能力都不能破坏现有找局、建局、入局、审核、沟通主链路；
5. V3 上线前需同步补齐任务执行计划、测试用例与验收文档。
