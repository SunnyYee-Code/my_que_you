# 雀友聚 V1 验收文档

## 1. 验收目标与范围

本文档用于基于 `memory-bank/v1/需求文档.md`、`memory-bank/v1/任务执行计划.md` 以及当前 `dev` 分支代码实现，对 `my_que_you` 的 V1 已交付范围进行结构化验收设计。

验收范围覆盖以下目标：

1. 验证 V1 需求文档中声明的全部业务模块是否存在明确实现落点；
2. 将 V1 需求拆解为可执行、可复现的验收场景；
3. 明确每个场景建议采用的验证方式：文档/代码走查、单元测试、集成测试、端到端测试、人工探索测试；
4. 为后续正式提测、回归和发布提供统一验收基线。

> 说明：本轮已删除仓库中的 `example` 示例测试，并继续补齐与 V1 测试用例直接对应的 Vitest / Playwright 测试代码。当前已形成一批可重复执行的页面集成测试与公开页 smoke 通过证据，但仍不能认定 V1 全量业务闭环已全部实测通过。

## 3. 验收策略

### 3.1 验收分层

为确保覆盖 V1 全量需求，采用以下四层验收：

1. **L1：静态验收 / 代码实现核对**
   - 核对需求是否有明确页面、Hook、上下文或数据库交互实现。
2. **L2：单元测试验收**
   - 适用于纯函数、状态判断、表单校验、文案规则、权限判断、违禁词判断、筛选排序等逻辑。
3. **L3：集成测试验收**
   - 适用于页面与 Hook 联动、按钮行为、跳转、条件渲染、数据状态流转等。
4. **L4：端到端验收**
   - 适用于跨页面主链路，例如“注册 → 引导 → 找局 → 申请 → 审核 → 群聊 → 评价”。

### 3.2 验收优先级

- **P0**：主业务闭环，失败则阻塞发布
- **P1**：核心辅助功能，失败会显著影响使用体验或治理能力
- **P2**：非阻塞体验项、展示项、弱依赖功能

## 6. 关键风险与验收关注点

### 6.1 当前高风险点

1. **测试基建薄弱**
   - 当前仓库几乎无正式测试用例，文档虽声明大量已实现能力，但自动化验证证据不足。
2. **强依赖 Supabase 在线数据与实时订阅**
   - 登录、好友、通知、群聊、私信、审核、后台等能力都依赖远端数据和实时能力，若无稳定测试数据，场景复现成本高。
3. **权限与状态流转组合复杂**
   - 房主/成员/游客/管理员在不同状态下的可见性和可操作性，需要重点验证边界条件。
4. **信用与治理规则易回归**
   - 退出扣分、踢人扣分、可申诉状态、举报/申诉后台可见性都属于高风险治理链路。
5. **违禁词过滤存在缓存机制**
   - `banned-words.ts` 采用 5 分钟缓存，需验证后台更新后缓存失效逻辑是否生效。

### 6.2 必测边界

- 未登录 / 已登录 / 管理员 / 超管 访问差异
- 未完成 onboarding 的准入拦截
- 已加入、待审核、已满员、已取消、已完成等状态切换
- 创建次数上限、参与次数上限
- 开局前 60 分钟边界退出
- 无定位权限 / 定位失败 / 无坐标拼团
- 邀请卡片已发送、已加入、已申请等重复状态
- 评价已提交 / 超时 / 重复评价
- 违禁词命中大小写、后台更新后缓存失效
